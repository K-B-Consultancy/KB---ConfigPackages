# TanStack Start — Stack Rules

> **Read [`../react-base/RULES.md`](../react-base/RULES.md) first.** This file covers only what's TanStack Start-specific.

## What TanStack Start adds over Vite

TanStack Start is a full-stack SSR framework built on TanStack Router. It adds to the Vite base:

- **Full-document SSR** — routes render on the server; the client hydrates
- **Server functions** (`createServerFn`) — type-safe RPC that runs server-only code called from anywhere
- **File-based routing** — same as TanStack Router, but with server-side rendering built in
- **Vinxi / Nitro** build — replaces the pure Vite SPA build; produces a Node.js server

Use TanStack Start (over plain Vite) when the app requires SSR, SEO-sensitive content, or server-side auth.

## Directory structure

```
src/
├── routes/
│   ├── __root.tsx         # Document shell (html/head/body), providers, error boundary
│   ├── index.tsx          # / route
│   ├── (auth)/            # Route group — shared layout, auth guard
│   │   ├── _layout.tsx
│   │   └── dashboard.tsx
│   └── api/               # Server-only API routes (non-HTML responses)
│       └── health.ts
├── features/              # Same features-based structure as react-base
│   └── <feature-name>/
│       ├── components/
│       ├── hooks/
│       └── api/
├── shared/
│   ├── components/        # Primitives, shadcn/ui components
│   ├── hooks/
│   └── lib/
├── server/
│   ├── functions/         # Shared server functions (called across multiple routes)
│   └── middleware/        # Global request middleware (auth, logging, CSRF)
├── router.tsx             # Router instance + QueryClient setup
├── client.tsx             # Client entry point
└── ssr.tsx                # Server entry point
```

`routeTree.gen.ts` is generated by TanStack Router's code-gen — **gitignored**, regenerated on `pnpm build` or `pnpm dev`.

## Routing

Routes are files under `src/routes/`. TanStack Router generates the type-safe route tree.

```tsx
// src/routes/index.tsx
import { createFileRoute } from '@tanstack/react-router'

export const Route = createFileRoute('/')({
  component: HomePage,
})

function HomePage() {
  return <div>Home</div>
}
```

### Route groups

Use parentheses for grouping without a URL segment:

```
src/routes/
├── (auth)/
│   ├── _layout.tsx        # Shared layout + auth guard for all routes in this group
│   └── dashboard.tsx      # /dashboard
└── (public)/
    └── login.tsx          # /login
```

### Dynamic segments

Use `$param` in the filename:

```
src/routes/users/$userId.tsx  →  /users/:userId
```

Access via `Route.useParams()` — fully type-safe. Always validate params with Zod via the `params` option on `createFileRoute`:

```ts
import { z } from 'zod'

export const Route = createFileRoute('/users/$userId')({
  params: {
    parse: (params) => ({ userId: z.string().uuid().parse(params.userId) }),
    stringify: ({ userId }) => ({ userId }),
  },
})
```

This coerces and validates at the router boundary, so `Route.useParams()` returns the parsed type rather than a raw string.

### Root route (`__root.tsx`)

The root route owns the document shell. This is where global providers go:

```tsx
import {
  createRootRoute,
  HeadContent,
  Outlet,
  Scripts,
} from '@tanstack/react-router'
import { QueryClientProvider } from '@tanstack/react-query'
import { queryClient } from '@/router'

export const Route = createRootRoute({
  head: () => ({
    meta: [
      { charSet: 'utf-8' },
      { name: 'viewport', content: 'width=device-width, initial-scale=1' },
    ],
  }),
  component: RootComponent,
})

function RootComponent() {
  return (
    <html lang="en">
      <head>
        <HeadContent />
      </head>
      <body>
        <QueryClientProvider client={queryClient}>
          <Outlet />
        </QueryClientProvider>
        <Scripts />
      </body>
    </html>
  )
}
```

Provider order inside `body`, outside-in: server-state client → monitoring → `<Outlet />`.

## Server functions

Server functions replace API routes for most backend logic. They run exclusively on the server and are callable from anywhere — components, hooks, other server functions.

```tsx
import { createServerFn } from '@tanstack/react-start'
import { z } from 'zod'
import { zodValidator } from '@tanstack/zod-adapter'

// In: src/server/functions/users.ts
export const getUser = createServerFn({ method: 'GET' })
  .validator(zodValidator(z.object({ id: z.string() })))
  .handler(async ({ data }) => {
    // Runs server-only. DB calls, secret env vars, etc. are safe here.
    return db.users.findUnique({ where: { id: data.id } })
  })
```

### Calling server functions

```tsx
// From a component (client-side)
const user = await getUser({ data: { id: userId } })

// From a route's beforeLoad (runs on server during SSR)
export const Route = createFileRoute('/users/$userId')({
  beforeLoad: async ({ params }) => {
    const user = await getUser({ data: { id: params.userId } })
    return { user }
  },
})
```

### Where server functions live

- **Route-specific** — define inline in the route file if used only there
- **Shared across routes** — define in `src/server/functions/<domain>.ts`
- **Never** import server functions in files that ship to the client bundle; TanStack Start tree-shakes them automatically, but circular imports can break this

### Middleware

Global middleware (auth checks, session loading, logging) goes in `src/server/middleware/`:

```tsx
// src/server/middleware/auth.ts
import { createMiddleware } from '@tanstack/react-start'

export const authMiddleware = createMiddleware().server(async ({ next, context }) => {
  const session = await getSession()
  if (!session) throw new Error('Unauthorized')
  return next({ context: { ...context, user: session.user } })
})
```

## Data fetching — TanStack Query

TanStack Query is the single source of truth for server state. Components use `useQuery` / `useSuspenseQuery`.

For SSR, data must be prefetched into the QueryClient before the server renders. Do this in a route's `loader` or `beforeLoad` — **the loader is used only for SSR prefetch, not as the primary data-management layer**. Components still own their data via Query hooks.

```tsx
// src/routes/users/$userId.tsx
import { createFileRoute } from '@tanstack/react-router'
import { useSuspenseQuery } from '@tanstack/react-query'
import { userQueryOptions } from '@/features/users/api/query-keys'

export const Route = createFileRoute('/users/$userId')({
  // Prefetch for SSR — data is available in the Query cache when the component renders
  loader: async ({ context, params }) => {
    await context.queryClient.ensureQueryData(userQueryOptions(params.userId))
  },
  component: UserPage,
})

function UserPage() {
  const { userId } = Route.useParams()
  // useSuspenseQuery — suspends during SSR until data is ready, no loading state needed
  const { data: user } = useSuspenseQuery(userQueryOptions(userId))
  return <div>{user.name}</div>
}
```

### QueryClient setup

The QueryClient is created once per request on the server and once on the client. Create it in `src/router.tsx`:

```tsx
// src/router.tsx
import { QueryClient } from '@tanstack/react-query'
import { createRouter } from '@tanstack/react-router'
import { routeTree } from './routeTree.gen'

export function createQueryClient() {
  return new QueryClient({
    defaultOptions: {
      queries: { staleTime: 60 * 1000 },
    },
  })
}

export function createAppRouter() {
  const queryClient = createQueryClient()
  return createRouter({
    routeTree,
    context: { queryClient },
    defaultPreload: 'intent',
  })
}
```

Pass `queryClient` through router context so route loaders can access it for prefetching.

## Authentication

Auth patterns vary per project. Common approaches used in KB Consultancy projects:

| Pattern | When to use |
|---|---|
| Session cookie + `createServerFn` | Default for most projects — server holds session, no token on client |
| Better Auth | When a full auth library with built-in providers is needed |
| Clerk | When social login, org management, or minimal custom code is the priority |

Document the chosen pattern in the app's `AGENTS.md` under "App-specific rules".

**Regardless of pattern:** auth checks always happen in server functions or middleware — never in client components. The client only receives what the server explicitly sends.

## Monitoring (Datadog)

For SSR apps, Datadog RUM initialises on the **client** only (same as Vite), but server-side logging goes through Datadog Logs via the Node agent or `dd-trace`.

- Service name: `{{CLIENT_SLUG}}-{{APP_NAME}}` — matches `VITE_APP_CLIENT` + app name (e.g., `acme-web`)
- **RUM:** initialize in a client-only component mounted in `__root.tsx`, guarded by `typeof window !== 'undefined'`
- **Server-side logs:** use `dd-trace` for tracing; write structured JSON logs to stdout — Coolify/Nixpacks forwards them to Datadog

## Build & deployment

### Local dev

```bash
pnpm dev   # SSR dev server with HMR (default port 3000)
```

No separate proxy config needed for local API calls — server functions run in the same process.

### Production build

```bash
pnpm build   # builds Node.js server to .output/
pnpm start   # runs the SSR server
```

The build produces a Node.js server (via Vinxi/Nitro). This is **not** a static export — unlike plain Vite apps, you need a Node runtime in production.

### Coolify deployment via nixpacks

TanStack Start produces a Node server, so nixpacks must start it (not serve static files):

```toml
# nixpacks.toml
[phases.install]
cmds = ["pnpm install --prod --frozen-lockfile"]

[phases.build]
cmds = ["pnpm install --frozen-lockfile", "pnpm build"]

[start]
cmd = "node .output/server/index.mjs"
```

The build phase re-runs `pnpm install` without `--prod` to bring in build-time dev dependencies (`vite`, `typescript`, `@tanstack/start`). Adjust the start command to match the actual output path (`pnpm build` will show it). The output path may vary across TanStack Start versions — verify after any framework upgrade.

## Common pitfalls

- **Running DB queries in client components** — they need to go inside a `createServerFn` handler, not in a hook
- **Importing server-only modules on the client** — e.g., `import { db } from '@/server/db'` in a component file. Use server functions to expose what the client needs
- **Skipping `ensureQueryData` in the loader** — the component will render without data on first SSR, causing a hydration mismatch or flash of empty content
- **Creating a new `QueryClient` inside a component** — one client per request/session; create it in `router.tsx` and pass via context
- **Forgetting `<Scripts />` in `__root.tsx`** — the client bundle won't load and hydration won't happen

## For AI agents

- When adding a new route, create the file under `src/routes/` — TanStack Router's code-gen picks it up automatically on the next build/dev start
- When a route needs data, add `ensureQueryData` in the loader for SSR and `useSuspenseQuery` in the component
- When adding server-only logic (DB, secrets, session), always put it inside `createServerFn` — never in a component or a shared utility that could be imported client-side
- When adding middleware (auth, logging), add it in `src/server/middleware/` and attach it to server functions with `.middleware([...])`
- Never use TanStack Router loaders as the canonical data store — they are only for SSR prefetch; the Query cache owns the data
