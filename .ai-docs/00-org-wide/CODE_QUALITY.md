# Code Quality and Static Analysis

This file defines the **org-wide policy** for code quality enforcement. The actual rule sets, formatters, and static analyzers are configured per stack — see `.ai-docs/10-stacks/<stack>/RULES.md`.

## The principle

Every stack must:

1. Have automated formatting that runs identically in **all environments** — developer IDE, terminal, AI agent session, CI
2. Have static analysis or a linter that catches issues at the **CLI level** (no IDE-only tools)
3. Block merge on any failure of (1) or (2)

The "all environments" rule rules out IDE-only solutions like SonarLint plugins, IntelliJ's built-in formatter, or VS Code-only setups. A rule that only fires inside one IDE is invisible to AI agents working from terminal or Copilot sessions, and gets bypassed by anyone using a different editor.

## Pre-commit hooks (lefthook)

Every project runs pre-commit hooks via **[lefthook](https://lefthook.dev/)** — single binary, no Node dependency, works for Java projects too. Hooks run on staged files only and call into the stack's own commands. The canonical hook config ships at `.ai-docs/config/lefthook.yml` — copy it to the repo root at init (`cp .ai-docs/config/lefthook.yml lefthook.yml && pnpm exec lefthook install`) and adjust only the app paths.

At minimum, the hook runs:

- Format check (scoped to staged files)
- Lint / static analysis (scoped to staged files)
- `git-secrets` scan to prevent committed credentials

## CI quality gates

Every PR runs, at minimum:

1. **Format check** — fail if any file would be reformatted
2. **Lint / static analysis** — `--max-warnings 0`; static analyzer reports must be empty
3. **Type check** — language-level type checking
4. **Test suite** — see [`TESTING.md`](./TESTING.md)
5. **Build** — production build must succeed

A failure in any of these blocks merge. No "merge override." Rule exceptions are made in a separate PR with reasoning, not by skipping the gate.

## Type system guardrails

Model finite state with discriminated unions (or equivalent tagged unions / sealed hierarchies), not optional boolean flags.

Optional booleans create impossible states (for example: "processing" and "paid" both true, or "failed" with a receipt ID). A discriminated union makes invalid states unrepresentable.

```typescript
// ❌ optional flags — "processing" and "paid" can both be true
interface Payment {
  isProcessing?: boolean;
  isPaid?: boolean;
  isFailed?: boolean;
  receiptId?: string;
}

// ✅ discriminated union — invalid combinations cannot be expressed
type Payment =
  | { status: "processing" }
  | { status: "paid"; receiptId: string }
  | { status: "failed"; error: string };
```

Type system guardrails reduce invalid AI-generated code by making bad states impossible to model.

## Architecture tests

Compiler checks catch only part of structural drift. Every codebase must also have architecture tests that enforce placement and dependency rules across layers — the concrete rule sets live in the stack implementations below, not here.

Architecture tests run in CI and block merge on failure. How each stack implements them:

- **Spring Boot** — ArchUnit tests enforcing the DDD layer dependencies; see [`spring-boot/RULES.md`](../10-stacks/spring-boot/RULES.md)
- **React stacks** — ESLint boundary rules (`no-restricted-imports`, `local/no-direct-query-in-components`, data-layer restrictions); see [`react-base/RULES.md`](../10-stacks/react-base/RULES.md) and [`nextjs/RULES.md`](../10-stacks/nextjs/RULES.md)

## Migration guardrails

Migrations generated by AI must be linted in CI. A migration that passes locally can still cause production incidents (for example table locks on high-row-count tables).

For any PR that touches a migrations folder, run a migration **safety** linter as a required check. PostgreSQL projects (the org default) use **Squawk**.

Safety linting is a different concern from SQL **style** linting (sqlfluff, in the cross-stack table below): Squawk blocks operationally unsafe patterns (non-concurrent indexes, dangerous renames, high-risk column changes); sqlfluff enforces consistent SQL formatting. A migrations PR runs both.

## Cross-stack tooling

For tooling that's truly cross-stack we standardize once, here:

**Shared rule sets ship as executable config files in `.ai-docs/config/`** (today: ESLint base + local rules, Prettier, lefthook), synced from KB-Documentation. Clients import them and never hand-write them; a rule is added or changed in KB-Documentation, never per project. Wiring per stack lives in the stack rules (e.g. `react-base/RULES.md` § ESLint and Prettier). Tools not yet shipped as configs (Stylelint, sqlfluff, tsconfig) are still configured per repo — see `docs/FUTURE_IMPROVEMENTS.md` #1 in KB-Documentation for the packaging plan.

**Enforced means deleted from prose.** The moment a rule is mechanically enforced — lint rule, formatter, ArchUnit or Error Prone check, CI guard, pre-commit hook — its prose form is removed from every rules file. What may remain is a one-line intent note linking the enforcing config or check. Code is the rule; docs keep only the why.

| Concern                      | Tool                                                  | Notes                                              |
| ---------------------------- | ----------------------------------------------------- | -------------------------------------------------- |
| CSS / SCSS lint              | [Stylelint](https://stylelint.io/)                    | Configured via `stylelint.config.cjs` at the repo root |
| SQL style lint               | [sqlfluff](https://www.sqlfluff.com/)                 | PostgreSQL dialect by default                      |
| SQL migration safety lint    | [Squawk](https://squawkhq.com/)                       | Blocks unsafe patterns; see Migration guardrails   |
| Secret scanning (pre-commit) | [git-secrets](https://github.com/awslabs/git-secrets) | Catches AWS, Stripe, GitHub tokens, etc.           |

Stack-specific tools (ESLint, Prettier, Spotless, Error Prone, etc.) are documented in the relevant stack rules.

## GitHub Copilot PR review

Every PR gets a Copilot review, invoked by the developer (in KB-Documentation itself it runs automatically). Copilot reads `.github/copilot-instructions.md` (which points to root `AGENTS.md`) and reviews the diff against the rules in `.ai-docs/`. Reviewer comments from Copilot are advisory; the merge requirement is the CI quality gates above.

## For AI agents

- **Run the format check and lint before declaring any task done.** A green test suite is not enough; format and lint are independent gates.
- **Never add a lint-disable comment to make code pass** (`// eslint-disable-next-line`, `@SuppressWarnings`). If the rule is wrong, change it in a separate PR. If the rule is right, fix the code.
- **Never lower a CI gate threshold to land a change.** Same principle.
- If you encounter a lint violation in code you didn't touch, leave it alone. Drive-by fixes create noisy diffs. Open a separate `chore` PR if needed.
